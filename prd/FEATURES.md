## 상세 기능 명세

### F1: 사진 업로드

- **유저 스토리**: "착장 사진을 가진 사용자로서, 사진을 빠르게 업로드하여 AI 분석을 시작하고 싶다."
- **인수 조건**:
  - [ ] 드래그앤드롭 + 파일 선택 두 방식 지원
  - [ ] JPG/PNG/WebP, 최대 10MB
  - [ ] 업로드 진행률 표시
  - [ ] Supabase Storage 업로드 성공 후 URL 반환
  - [ ] 모바일 브라우저에서 카메라 직접 촬영 지원 (input capture)
  - [ ] 업로드 후 사진 미리보기 표시
- **성공 시나리오**: 5MB JPG 드래그 → 3초 내 업로드 → 미리보기
- **실패 시나리오**:
  - 10MB 초과 → "사진은 10MB 이하만 가능합니다" 토스트
  - 네트워크 실패 → 재시도 버튼
- **뒷받침 인사이트**: USERS.md — "작심삼일 페인", 첫 마찰을 최소화해야 함

---

### F2: AI 아이템 분석

- **유저 스토리**: "착장 사진을 업로드한 사용자로서, 각 아이템(상의/하의/신발 등)이 자동으로 식별되고 설명되기를 원한다."
- **인수 조건**:
  - [ ] Claude Vision 또는 GPT-4o Vision으로 분석
  - [ ] 카테고리 분류: 상의, 하의, 아우터, 신발, 가방, 액세서리 (최소 6개)
  - [ ] 각 아이템: 색상, 스타일 키워드 2-3개, 설명 1문장 반환
  - [ ] 전체 스타일 요약 1-2문장 (예: "조용한 럭셔리 + 빈티지 믹스")
  - [ ] 해시태그 3-5개 자동 생성
  - [ ] 분석 시간 < 10초 (P95)
  - [ ] 응답은 JSON 스키마 강제 (structured output)
- **성공 시나리오**: 사진 1장 → 7초 → 아이템 5개 + 요약 + 해시태그
- **실패 시나리오**:
  - 사람이 안 보이는 사진 → "착장이 명확히 보이지 않습니다. 다시 시도해주세요"
  - API 타임아웃 (15초) → 재시도 옵션
  - JSON 파싱 실패 → 1회 자동 재시도
- **뒷받침 인사이트**: 페인포인트 리서치 — "수동 태깅 10-15분" / Round 3 — AI 정확도 70-80% 가정 + 수정 UI 필수

### F3: AI 손글씨 카드 생성 (Plan A — gpt-image-1)

- **유저 스토리**: "분석 완료한 사용자로서, 원본 사진 위에 손글씨로 아이템 정보가 그려진 멋진 카드를 생성하고 싶다."
- **인수 조건**:
  - [ ] OpenAI gpt-image-1 API 호출 (image edit endpoint)
  - [ ] 입력: 원본 사진 + 분석 결과 + 손글씨 프롬프트
  - [ ] 출력: 원본 사진 위에 화살표/주석이 그려진 카드 이미지 (1080x1920 권장)
  - [ ] 생성 시간 < 30초 (P95)
  - [ ] 생성 후 카드 미리보기 + 재생성 버튼 (재생성 시 카운트 차감)
  - [ ] 생성된 이미지는 Supabase Storage에 저장
  - [ ] Plan A 실패 자동 감지 시 Plan B로 자동 fallback (사용자에게 알림)
- **성공 시나리오**: 분석 완료 → 카드 생성 클릭 → 25초 → 손글씨 카드 표시 → 공유 페이지로 이동
- **실패 시나리오**:
  - API 오류 → Plan B 자동 전환
  - 결과 이미지가 부적절 → 재생성 버튼 (무료 차감)
  - 사용량 한도 초과 → 유료 안내 모달
- **뒷받침 인사이트**: PRD 인사이트 #3 — 최대 기술 위험. 핵심 차별화.

### F3-B: html2canvas 카드 생성 (Plan B — Fallback)

- **유저 스토리**: (Plan A 실패 시 자동 진입) "사용자로서, AI 카드를 받을 수 없을 때라도 정보 카드를 받고 싶다."
- **인수 조건**:
  - [ ] React 컴포넌트로 카드 레이아웃 렌더링
  - [ ] Google Fonts "Nanum Pen Script" 적용 (한국어 손글씨)
  - [ ] 원본 사진 위에 SVG 화살표 + 텍스트 오버레이
  - [ ] html2canvas로 PNG 변환
  - [ ] 1080x1920 비율 고정
  - [ ] 생성 시간 < 5초 (서버사이드)
  - [ ] PNG → Supabase Storage 저장
- **성공 시나리오**: Plan A fail → Plan B 진입 → 3초 → 카드 표시
- **실패 시나리오**: 폰트 로딩 실패 → 시스템 폰트로 fallback
- **뒷받침 인사이트**: PRD Hat 충돌 해소 — Yellow vs Black

### F4: 공유 페이지 (/share/[id])

- **유저 스토리**: "카드를 생성한 크리에이터로서, 카드와 아이템 정보를 URL 하나로 팔로워에게 전달하고 싶다."
- **인수 조건**:
  - [ ] Next.js 동적 라우트 `/share/[share_id]`
  - [ ] SSR 렌더링 (SEO + OG 메타태그)
  - [ ] OG 이미지 = 카드 이미지 (인스타/카카오톡 미리보기 최적화)
  - [ ] 공개 페이지: 카드 + 아이템 리스트 + 작성자 닉네임
  - [ ] 비공개 OOTD는 404
  - [ ] 비로그인 사용자도 조회 가능
  - [ ] "이 카드 만들기" CTA (회원가입 유도)
  - [ ] URL 복사 버튼
  - [ ] 모바일 반응형
- **성공 시나리오**: 크리에이터가 URL을 인스타 바이오에 박음 → 팔로워 클릭 → 카드 + 아이템 정보 확인
- **실패 시나리오**:
  - 잘못된 share_id → 404 페이지
  - 비공개 OOTD → 404
- **뒷받침 인사이트**: PRD 인사이트 #1 — 제품 핵심이 "공유". MVP 핵심 차별화.

### F5: 소셜 로그인 (Google + Kakao)

- **유저 스토리**: "신규 사용자로서, 비밀번호 없이 빠르게 가입하고 싶다."
- **인수 조건**:
  - [ ] NextAuth.js 사용
  - [ ] Google OAuth Provider
  - [ ] Kakao OAuth Provider (커스텀)
  - [ ] 첫 로그인 시 users 테이블에 자동 레코드 생성
  - [ ] 기본 plan = "free"
  - [ ] 세션 7일 유지
- **성공 시나리오**: Google 클릭 → 3초 → 로그인 완료
- **실패 시나리오**: OAuth 거부 → 로그인 페이지로 리다이렉트
- **뒷받침 인사이트**: USERS.md — 페르소나 2(민지) "가입 절차 복잡" 이탈 트리거

### F6: OOTD 기록 저장

- **유저 스토리**: "카드 생성한 사용자로서, 내 OOTD 기록을 저장하여 나중에 다시 볼 수 있기를 원한다."
- **인수 조건**:
  - [ ] 카드 생성 후 자동 저장 (날짜 기본값 = 오늘)
  - [ ] 사용자 날짜 변경 가능
  - [ ] 메모 필드 (선택, 200자)
  - [ ] 공개/비공개 토글 (기본값 = 비공개)
  - [ ] 저장 후 "내 OOTD" 페이지로 이동
- **성공 시나리오**: 카드 생성 → 자동 저장 → 토스트 "저장 완료"
- **실패 시나리오**: DB 저장 실패 → 재시도 버튼 + 로컬 임시 저장

### F7: 캘린더 리스트 뷰

- **유저 스토리**: "기록형 사용자로서, 월별 OOTD를 한눈에 보고 싶다."
- **인수 조건**:
  - [ ] 월 단위 페이지네이션 (이전/다음 월 버튼)
  - [ ] 날짜순 정렬 (최신 위)
  - [ ] 각 항목: 썸네일 + 날짜 + 스타일 요약 1줄
  - [ ] 항목 클릭 → 상세 페이지
  - [ ] MVP에서는 리스트 형식 (그리드/캘린더 풀뷰는 Should)
- **성공 시나리오**: "내 OOTD" 진입 → 2025년 5월 리스트 표시

### F8: AI 분석 결과 편집 UI

- **유저 스토리**: "AI 분석 결과가 틀린 사용자로서, 잘못된 카테고리/색상/설명을 직접 수정하고 싶다."
- **인수 조건**:
  - [ ] 분석 결과 화면에서 각 아이템 카드를 클릭 → 편집 모드
  - [ ] 편집 가능 필드: 카테고리(드롭다운), 색상, 설명, 브랜드, 가격, 구매 URL
  - [ ] 아이템 삭제 / 추가 가능
  - [ ] 편집 후 "이대로 카드 만들기" 버튼
  - [ ] 편집 이벤트 로깅 (AI 수용률 측정용)
- **성공 시나리오**: AI가 신발을 가방으로 잘못 분류 → 클릭 → 카테고리 변경 → 저장
- **뒷받침 인사이트**: USERS.md — 사용자는 70% 정확도여도 사용함, 단 수정 쉬워야 함

### F9: 사용량 제한 (월 5회/30회)

- **유저 스토리**: "비즈니스 관점에서, 카드 생성 API 비용을 사용자 플랜에 따라 제한해야 한다."
- **인수 조건**:
  - [ ] usage_logs 테이블에 year_month, card_generation_count 기록
  - [ ] 카드 생성 API 호출 전 카운터 체크
  - [ ] free 플랜: 월 5회 초과 시 차단 + 유료 안내 모달
  - [ ] pro 플랜: 월 30회 초과 시 차단 + 한도 안내
  - [ ] 매월 1일 00:00 KST 카운터 리셋 (cron 또는 lazy reset)
  - [ ] 사용자 대시보드에 "이번 달 N/5회 사용" 표시
- **성공 시나리오**: 무료 사용자가 6회째 시도 → 모달 "월 한도 초과, 유료 전환 시 30회"
- **실패 시나리오**: cron 실패 → lazy reset 로직 (요청 시점에 년월 비교)

- **유저 스토리**: "프라이버시를 중시하는 사용자로서, 어떤 OOTD를 공개할지 선택하고 싶다."
- **인수 조건**:
  - [ ] 기본값 = 비공개 (is_public = false)
  - [ ] 저장/편집 화면에서 토글
  - [ ] 공개 시 share_id 생성 (nanoid 8자)
  - [ ] 비공개 OOTD는 /share/[id] 접근 불가 (404)
  - [ ] 비공개 ↔ 공개 전환 가능
